## 3. Анализ стиля

| Срез | Оценка |
|---|---|
| **Доминирующий стиль** | игровой + мемный с локальной мифологией «Возни» (аптека/схема/облава, Радик, Конинг, Зволле, Burger King, гэмблинг-юмор, мат) |
| **Единообразие** | высокое внутри геймплея (farm/duel/treasure/casino/pidor/para — единый мемный голос, пулы вариантов). Социалка (marriage) — игровая, но без вариативности. |
| **Технический «бот»-стиль** | админка, привязка (OIDC), часть ошибок и системных колбэков — сухие, без персоны |
| **Повторы** | низкие: благодаря пулам `*_VARIANTS` геймплей не повторяется. Шаблоны через `{mention}/{amount}/{time}` единообразны. |
| **Готовность к персоне** | геймплей уже «звучит» как Возня; технические зоны (привязка/админка/ошибки) — нет |

Вывод: проект НЕ выглядит как голый технический бот в игровой части — там уже есть
характер. «Технический» вид остаётся в служебных зонах (привязка, админка, часть
ошибок) и в нейтральных карточках (профиль, баланс, MMR, репутация).

## 5. Рекомендации по централизации

1. **Базис уже хороший.** `app/settings/texts.py` фактически выполняет роль
   localization layer для геймплея — 70+ констант, редактируемых не-программистом
   (об этом прямо сказано в шапке файла). Это сильная стартовая точка.
2. **Свести остатки в `settings/`.** Вынести 4 инлайн-строки привязки, 2 строки
   marriage, 2–3 строки achievements/service, инлайн-части профиля. После этого
   100% пользовательского текста будет в одном слое.
3. **Единый namespace.** Сейчас тексты раскиданы по 5 файлам в `settings/`
   (`texts.py`, `reputation.py`, `mmr.py`, `titles.py`, `achievements.py`). Для
   полной локализации стоит ввести единый реестр ключей (например, `TEXTS["farm.gain"]`)
   или хотя бы единый модуль-агрегатор, чтобы будущий перевод/персона работали с
   одним источником.
4. **Описания команд** (`BOT_COMMANDS` в `main.py`) тоже user-facing — включить в
   локализуемый слой.
5. **Локаль как параметр.** Если планируется многоязычность — ввести функцию
   `t(key, lang, **kwargs)` вместо прямого обращения к константам. Если язык один
   (только русский + персона Возни) — достаточно централизации без locale-измерения.

### Самые сложные для миграции системы

| Сложность | Система | Почему |
|---|---|---|
| 🔴 высокая | Профиль | текст собирается из частей прямо в коде, не отдельные константы |
| 🔴 высокая | Достижения | 27×2 строк + инлайн-фрагменты в service; смешанный источник |
| 🟡 средняя | Привязка (OIDC) | 4 инлайн-строки, но в чувствительном auth-флоу — менять осторожно |
| 🟢 низкая | Геймплей (farm/duel/treasure/casino/pidor/para) | уже 100% в `texts.py`, только переименовать ключи |
| 🟢 низкая | Команды бота | один список в `main.py` |

## 6. Кандидаты на систему вариативных ответов

Уже вариативные (пулы `*_VARIANTS` в `texts.py`) — менять не нужно, расширять можно:
- `FARM_GAIN_VARIANTS` (22), `FARM_ZERO_VARIANTS` (6), `FARM_LOSS_VARIANTS` (18)
- `WELCOME_VARIANTS`
- казино (CASINO win/loss/jackpot пулы)
- клад (TREASURE/CLAD spawn + claim пулы)
- дуэль (фраза-пул результата)

Всегда одна фиксированная фраза (кандидаты на будущую вариативность):
- **Свадьба:** `MARRY_DONE`, `MARRY_PROPOSAL`, `DIVORCE_DONE` — по одной фразе.
- **Пара дня / Пидор дня:** объявление победителя — фиксировано.
- **Репутация:** карточка и подтверждение — фиксировано.
- **MMR:** карточка и ап-ранга — фиксировано.
- **Профиль/баланс:** карточки — фиксированы (это нормально, данные-карточки).
- **Кулдаун** `COOLDOWN_NOTICE` — одна фраза на все системы.
- **Ошибки** `ERROR_GENERIC`, `USER_NOT_FOUND` — фиксированы.

Приоритет вариативности (по «весу» в общении): свадьба/развод → пара дня → пидор
дня → ап MMR-ранга → кулдаун. Эти моменты эмоциональные и выиграют от пула фраз.
