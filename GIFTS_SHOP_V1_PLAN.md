# Gifts Shop V1 — аудит и план реализации

Цель: первый полный пользовательский цикл **игрок → ешки → покупка Gift →
выдача Gift**, полностью аудируемый, с видимыми выручкой/себестоимостью/P&L.

> Сначала аудит и план. Реализация — после согласования.
> Принципы: бот — единственный писатель; без новых фундаментальных таблиц, если
> не абсолютно необходимо; каждая покупка аудируема; экономика видит P&L.

Дата: 2026-06-07. HEAD: `0020_seed_gift_catalog`. Курс: 1★≈10 ешек.

---

## Часть 1. Аудит существующих сущностей

### 1.1. `gift_catalog` (миграция 0018) — товар
Поля: `id, code, name, description, star_cost, price_eshki, telegram_gift_id,
stock, reserved, sold_count, is_active, sort_order, meta, created_at, updated_at`.
- **Готово для магазина:** цена в ешках (`price_eshki`), себестоимость в Stars
  (`star_cost`), запас (`stock`), резерв (`reserved`), счётчик продаж
  (`sold_count`) — всё под поток покупки уже есть.
- Сейчас заполнен сидом 0020 (5 позиций). CHECK уже защищает `reserved ≤ stock`.
- **Вывод: готов полностью, изменений не нужно.**

### 1.2. `purchase_history` (миграция 0010) — запись покупки (деньги)
Поля: `id, user_id, offer_id (int), item_code, price, quantity, source,
transaction_id, audit_id, meta, created_at`.
- **Идеально подходит как журнал покупок Gifts:** `source='gift'`,
  `offer_id = gift_catalog.id`, `item_code = gift code`, `price = price_eshki`,
  `transaction_id` = id списания ешек, `meta = {star_cost, ...}`.
- Есть частичный уникальный индекс `uq_purchase_user_offer_unique` на
  `(user_id, offer_id) WHERE meta->>'unique'='true'` — для Gifts **НЕ** ставим
  `meta.unique` (подарки покупаются многократно).
- **Вывод: переиспользуем как есть, без изменений.**

### 1.3. `gift_transactions` (миграция 0011) — журнал передачи актива + СТАТУС
Поля: `id, kind, gift_type, sender_user_id, recipient_user_id, item_code,
quantity, amount, status, idempotency_key (unique), transaction_id, audit_id,
meta, created_at`.
- Спроектирован для player↔player передач (kind item/currency), НО содержит ровно
  то, что нужно для **жизненного цикла выдачи**:
  - `status` ∈ (`completed`, `pending`, `cancelled`) — статус выдачи;
  - `idempotency_key` UNIQUE — **защита от двойной выдачи**;
  - `gift_type='system'` — выдача от системы игроку;
  - `transaction_id` — связь со списанием ешек;
  - `meta` — место для `star_cost`, `gift_code`, результата Telegram API.
- **Тонкое место:** CHECK `ck_gift_kind_payload` разрешает только
  `kind='item'`+item_code или `kind='currency'`+amount. Реальный Telegram Gift —
  это не предмет инвентаря и не ешки. Два варианта (см. §3.минимальные изменения).
- **Вывод: переиспользуем как журнал доставки**; нужна одна узкая правка CHECK
  (не новая таблица).

### 1.4. `transactions` (0001) — леджер ешек
- Списание за покупку идёт через `change_balance_tx()` (уже есть после L1):
  `reason='purchase'`, `meta={source:'gift_buy', gift:<code>}`, возвращает
  проводку с `id` → кладём в `purchase_history.transaction_id` и в
  `gift_transactions.transaction_id`. **Готово, изменений нет.**

### 1.5. `inventory` / `inventory_items`
- Для **реального** Telegram Gift инвентарь НЕ задействуется: подарок уходит во
  «внешний мир» (Telegram), а не на полку игрока. `inventory` остаётся для
  косметики/предметов. **Не трогаем.**

### 1.6. Admin gifts (`/api/admin/gifts`, `/admin/gifts`)
- Управление каталогом (CRUD, цена, запас, себестоимость) — готово. Покупки/выдачи
  пока нет. Витрина игрока (`/cases` есть для кейсов; для gifts витрины нет).
- **Вывод: каталог-админка готова; нужен пользовательский магазин (бот) +
  отображение статусов выдачи в админке.**

### Сводка аудита
| Сущность | Роль в магазине | Готовность | Изменения |
|---|---|---|---|
| `gift_catalog` | товар, цена, запас, себестоимость | ✅ полная | нет |
| `purchase_history` | запись покупки (деньги) | ✅ полная | нет (без meta.unique) |
| `gift_transactions` | журнал доставки + статус + идемпотентность | ⚠️ почти | 1 правка CHECK |
| `transactions` + `change_balance_tx` | списание ешек | ✅ полная | нет |
| `inventory*` | — | n/a | не трогаем |
| admin gifts | каталог | ✅ | + экран статусов (read-only) |

**Новые фундаментальные таблицы НЕ нужны.** Весь цикл ложится на существующее.

## Часть 2. Дизайн магазина

### 2.1. Пользовательский магазин Gifts (бот)
- Команда `/подарки` (и `/gifts`): показывает активные позиции `gift_catalog`
  (`is_active`, `stock − reserved − sold_count > 0`), цену в ешках, остаток.
- Inline-кнопка «Купить за N ешек» под каждой позицией → callback покупки.
- Баланс игрока показывается; недостаток ешек → понятный отказ.

### 2.2. Процесс покупки за ешки (атомарно, одна сессия БД)
Аналогично `open_case()` — pre-flight под блокировками, потом мутации:
1. **Pre-flight (FOR UPDATE):** заблокировать строку `gift_catalog` и `users`;
   проверить: позиция активна, есть остаток (`stock IS NULL OR
   reserved+sold_count < stock`), баланс ≥ `price_eshki`. Любой отказ → ранний
   возврат без мутаций.
2. **Списать ешки:** `change_balance_tx(-price_eshki, reason='purchase',
   meta={source:'gift_buy', gift:code})` → получить `transaction_id`.
3. **Зарезервировать:** `gift_catalog.reserved += 1` (резерв «занял» единицу пула
   до подтверждения выдачи).
4. **Создать запись покупки:** `purchase_history(source='gift', offer_id=id,
   item_code=code, price=price_eshki, quantity=1, transaction_id, meta={star_cost})`.
5. **Создать запись доставки:** `gift_transactions(gift_type='system',
   recipient_user_id=buyer, status='pending', idempotency_key=<purchase key>,
   transaction_id, meta={gift_code, star_cost})`.
6. Коммит middleware. Игроку: «Подарок оплачен, отправляю…».

Денежная часть закоммичена ДО внешнего вызова Telegram (деньги списаны, резерв
занят, доставка `pending`). Внешняя отправка — отдельным шагом (§2.4), потому что
Telegram API нельзя звать внутри транзакции БД.

### 2.3. Резервирование подарка
- `reserved` в `gift_catalog` растёт при покупке (шаг 3) и держит место в пуле,
  пока доставка `pending`. Это потолок расхода Stars: продать сверх `stock`
  нельзя (CHECK `reserved ≤ stock` + проверка в pre-flight).
- При успешной выдаче: `reserved -= 1`, `sold_count += 1` (единица «реализована»).
- При отмене/возврате: `reserved -= 1` (место возвращается в пул).

### 2.4. Выдача подарка (внешний шаг, вне транзакции БД)
- Воркер/хендлер берёт `gift_transactions` со `status='pending'` и зовёт Telegram
  API отправки подарка (`sendGift` / соответствующий метод).
- **Успех:** в новой транзакции БД — `status='completed'`,
  `meta.telegram_charge_id`/`meta.result`, `gift_catalog`: `reserved-1`,
  `sold_count+1`. Игроку: «Подарок отправлен 🎁».
- **Идемпотентность выдачи:** перед отправкой повторно читаем строку
  `FOR UPDATE`; если уже `completed` — выходим (не шлём второй раз). Плюс
  `idempotency_key` UNIQUE не даст создать дубль записи доставки.

### 2.5. Отмена и возврат
- Триггеры: Telegram API вернул неустранимую ошибку, позиция исчезла, ручная
  отмена админом.
- В одной транзакции: `gift_transactions.status='cancelled'` (+ `meta.error`);
  **вернуть ешки** `change_balance_tx(+price_eshki, reason='reward',
  meta={source:'gift_refund', gift:code, of_transaction:<orig tx id>})`;
  `gift_catalog.reserved -= 1` (место назад в пул). Запись покупки в
  `purchase_history` остаётся (помечаем `meta.refunded=true`) — история не
  переписывается.
- Игроку: «Не удалось отправить подарок, ешки возвращены».

### 2.6. Учёт себестоимости в Stars
- `star_cost` фиксируется снапшотом в `purchase_history.meta.star_cost` и
  `gift_transactions.meta.star_cost` в момент покупки (цена/курс могут меняться —
  историческая себестоимость не теряется).
- **Фактический расход Stars = Σ `star_cost` по `gift_transactions` со
  `status='completed'`** (подарок реально отправлен → Stars потрачены).
- Полноценный `stars_ledger` (донат + расход одной строкой) — отложен; для V1
  расход Stars берём из завершённых доставок (этого достаточно для P&L Gifts).

### 2.7. Учёт прибыли/убытка (P&L Gifts)
Economic Control Center → вкладка Gifts начинает показывать реально:
- **Выручка (ешки):** Σ `purchase_history.price` где `source='gift'` (за вычетом
  возвратов).
- **Реализованный расход Stars:** Σ `meta.star_cost` по `completed`-доставкам.
- **Маржа в ешках:** выручка − Σ(`star_cost`×10) по завершённым (наценка).
- **Воронка статусов:** pending / completed / cancelled (контроль зависаний).
- **Баланс «подарочного фонда»** в Stars остаётся приблизительным до
  `stars_ledger` (донат-приход пока не логируется) — показываем расход, помечаем
  приход как «—».

## Часть 4. Первый реальный Telegram Gift — детали

- **Где хранится статус:** `gift_transactions.status` (`pending` → `completed` /
  `cancelled`), плюс детали в `meta` (`telegram_charge_id`, `error`).
- **Как избежать двойной выдачи:**
  1. `idempotency_key` UNIQUE на запись доставки (один ключ — одна доставка);
  2. перед внешней отправкой — `SELECT ... FOR UPDATE` строки доставки; если уже
     `completed` → выходим;
  3. перевод в `completed` и внешняя отправка сериализованы блокировкой строки.
- **Как обрабатывать ошибки Telegram API:**
  - временная ошибка (сеть, лимит) → оставляем `pending`, ретрай позже
    (ограниченное число попыток, в `meta.attempts`);
  - неустранимая ошибка (нельзя отправить этому юзеру) → `cancelled` + возврат
    ешек (§2.5);
  - неоднозначный ответ → НЕ помечаем `completed`, ждём ручной разбор (доставка
    видна в экране статусов).
- **Как компенсировать неудачу:** автоматический возврат ешек
  (`change_balance_tx(+price)`, `reason='reward'`, `meta.source='gift_refund'`) +
  возврат места в пул (`reserved-1`). Деньги игрока не теряются ни при каком
  исходе.
