## 6. Производительность

**Главная проблема — MMR.** `app/repositories/mmr.py:get_mmr` агрегирует
`SUM(amount)` по всему журналу `mmr_entries` при каждом вызове. Кеша текущего
значения на `users` нет. Следствия:
- `duel/service.py:accept_challenge` — `award_mmr` дважды (победитель+проигравший),
  каждый делает SUM до и после → ~4 агрегации на один бой.
- `achievements/service.py` — `award_mmr` внутри цикла выдачи ачивок → по SUM
  на каждую выданную ачивку.
- Профиль и `топммр` — снова полные агрегации.

Индексы `ix_mmr_player`, `ix_mmr_player_created` есть — агрегат покрыт, но
**сам факт пересчёта журнала на каждый чих** не масштабируется.

**Фикс:** денормализовать текущий MMR в `users.mmr` (или отдельную проекцию),
журнал оставить как audit. Обновлять инкрементально при `award_mmr`. Это
снимет 90% MMR-нагрузки. Прочие фичи (профиль, рейтинги, репутация) —
single-query, проблем роста не выявлено.

## 9. Синхронизация бота и сайта — точки дублирования

| Данные | В боте | На сайте | Риск |
|---|---|---|---|
| MMR-ранги | `app/settings/mmr.py: RANKS` | `lib/mmr.ts: MMR_RANKS` | ручная синхра (зафиксировано в AGENTS.md) |
| Титулы/звания | `app/settings/*` | `lib/queries.ts` / live-компоненты | дубль порогов титулов |
| Команды бота | роутеры | `lib/voznya-bot.ts: COMMAND_GROUPS` | ручная синхра списка команд |
| Ачивки | `app/settings` ACHIEVEMENTS | live achievements-catalog | дубль определений |
| Экономика-константы | `app/settings/balance.py` | hardcode в компонентах | дубль |

**Решение (архитектурное):** один источник правды. Варианты:
- (A) Бот экспонирует read-only catalog-API (`/internal/catalog/*`), сайт тянет.
- (B) Общий JSON-каталог в репозитории, который обе стороны читают при сборке.
Рекомендую (A) для динамики (ранги/титулы/ачивки) — сайт и так читает БД бота.

## 11. Будущая экономика — оценка архитектуры

- **Кейсы** (обычные/редкие/сезонные/донатные): новая таблица `cases` +
  `case_drops` (item_code, weight). Открытие — атомарная транзакция в боте
  (бот — единственный писатель gameplay-таблиц). Шансы редактируются из админки.
- **Telegram Gifts** (покупка/продажа/обмен): использовать существующую
  foundation `gifts` + `inventory`. Обмен — через pending_actions-подобный
  flow с подтверждением.
- **Коллекции**: таблица `collections` + связь с item types, бонус за полный сбор.
- **Косметика** (титулы/рамки/бейджи/витрины): тип предмета в inventory +
  поле «экипировано» на профиле. Витрина профиля = выбранные предметы.

Принцип: бот пишет, сайт показывает и админит. Кейсы/обмены — только в боте.
