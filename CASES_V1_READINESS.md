## 1. Что готово (реализовано и проверено чтением кода)

### Бэкенд-ядро (бот)
- **Миграция 0016** `0016_cases_foundation` — 4 таблицы
  (`inventory_instances`, `case_definitions`, `case_rewards`, `case_openings`) +
  аддитивные поля каталога (`stackable`, `collection_code`, `series_total`).
  Полный `downgrade()`. Линейная цепочка: `down_revision =

- **Модели** SQLAlchemy: `CaseDefinition`, `CaseReward`, `CaseOpening`,

  индексы совпадают с миграцией.
- **Репозиторий** `app/repositories/cases.py`: чтения + `get_available_rewards_
  for_update` (FOR UPDATE, отсекает исчерпанные лимиты).
- **`grant_reward()`** — единая точка выдачи (`item` → `grant_item`,
  `currency` → `change_balance`; `tg_gift`/`stars` → `NotImplementedError`).
- **`open_case()`** — единственная атомарная точка открытия. **Переписана**:
  теперь pre-flight под блокировками (см. §6, fix критической дыры).
- **Хендлеры бота**: `/кейсы`, `/кейс <code>`, `/открыть <code>` + callback
  `case:open:<code>:<user_id>` (проверка владельца кнопки).
- **Тексты** кейсов в `app/settings/texts.py`; кнопка `case_open` в keyboards.
- **Тест** `tests/test_cases_pick_reward.py` — распределение весов `_pick_reward`.

### Админка + сайт (v0-voznya)
- **Права `cases.*`** в синхроне Python (`app/core/permissions.py`) ↔ TS
  (`lib/auth/admin-permissions.ts`): `cases.view` (support+),
  `cases.manage`/`cases.grant` (admin+).
- **Admin API**:
  - `GET/POST /api/admin/cases` — список + создать/обновить определение
    (валидирует, что `item_code` есть в каталоге и имеет `type='case'`);
  - `GET/POST/DELETE /api/admin/cases/[code]/rewards` — дроп-лист (V1 только
    `item`/`currency`, отклоняет `tg_gift`/`stars`);
  - `GET /api/admin/cases/openings` — леджер открытий (фильтры user/case).
  Все мутации — в транзакции + строка `audit_log`.
- **Admin UI** `/admin/cases`: список кейсов, форма создания/редактирования,
  редактор дроп-листа с живыми шансами (`weight/Σweight`), ItemPicker по
  каталогу. Пункт «Кейсы» в навигации админки.
- **Публичная витрина** `/cases`: активные кейсы, содержимое, редкости, честные
  шансы. Read-only через `lib/cases.ts`. Мягкая деградация до миграции.

- **Коллекции** (`collection_code`/`series_total`) — поля есть, логики нет.
- **Достижения по кейсам** — есть хук `emit_case_opened`, но проверок ачивок

- **MMR-награды из кейсов** — не в V1.
- **Mini App** — отдельный план (`MINI_APP_FOUNDATION.md`), не реализуется.
- **Анимация открытия на сайте** — витрина read-only; открытие пока только в
  боте (по решению: FastAPI-слой вводится на этапе Mini App).
- **`/cases`,`/open` в `BOT_COMMANDS`** (`app/main.py`) — сознательно не
  добавлены до согласования меню.
- **Выдача кейса игроку из админки** — отдельной кнопки «выдать кейс» нет;
  делается через существующую item-выдачу (`/api/admin/inventory`, выдать
  предмет с `type='case'`). Это и есть «выдать кейс» — отдельный роут не нужен.

## 5. Ручная проверка (чек-лист перед production)

**Подготовка данных** (после миграции):
1. Создать в каталоге предмет с `type='case'` (например `case_rookie`) и
   предмет-награду (например титул/бейдж) — каталог ведёт бот; сейчас items
   заводятся через прямой SQL/админ-выдачу. (Заведение item-каталога — отдельный
   вопрос, не часть Cases V1.)
2. В `/admin/cases`: создать кейс на `case_rookie`, добавить 2–3 дропа
   (currency + item), проверить, что сумма шансов = 100%.

**Открытие в боте:**
3. `/кейсы` — кейс виден, стоимость и «у тебя: N» корректны.
4. `/кейс case_rookie` — карточка, дроп-лист с шансами, кнопка «Открыть».
5. Выдать себе 1 кейс (admin item grant), `/открыть case_rookie` —
   награда выдана, кейс списан, в `case_openings` строка с `roll`/`weight_
   snapshot`, баланс/инвентарь изменились.
6. Попытка открыть без кейса → «нет кейса», ничего не списано.
7. Кейс с `open_cost_kind=currency`: при нехватке ешек → «не хватает», **ключ
   НЕ списан** (критично — см. §6), баланс не тронут.
8. Чужая кнопка: другой пользователь жмёт «Открыть» → «не твой кейс».
9. Лимитка (`max_global_supply=1`): выпадает один раз, далее исключается.

**Сайт:**
10. `/cases` (без логина) — витрина, шансы совпадают с админкой.
11. `/admin/cases` под ролью support — только просмотр (нет форм); под admin —
    CRUD работает; каждое действие пишет строку в `/admin/audit`.

**Откат:**
12. `alembic downgrade -1` проходит без ошибок (на dev).

## 6. Найденные проблемы и что исправлено в этой итерации

### КРИТИЧНО (исправлено): потеря ключа при платном открытии
`DbSessionMiddleware` коммитит сессию при ОБЫЧНОМ возврате хендлера и
откатывает только при исключении. Прежняя `open_case` списывала ключ
(`consume_item`), затем списывала ешки, и при `InsufficientFunds`
**возвращала** `OpenResult(status="not_enough")` — обычный возврат →
middleware **коммитил** уже списанный ключ. Для кейса с `consumes_key=true` и
`open_cost_kind='currency'` игрок с ключом, но без ешек, **терял кейс и не
получал ничего**.

Исправление (`app/features/cases/service.py`): введён **pre-flight** — сначала
под блокировками строк (`FOR UPDATE` на владении ключом и на строке игрока)
проверяется наличие ВСЕХ ресурсов БЕЗ мутаций; неуспех возвращается до первой
записи. Мутации (списание ключа, ешек, инкремент лимита, выдача, леджер)
выполняются только когда отказ уже невозможен. Блокировки держатся до конца
транзакции, поэтому фактические списания не могут отказать.

### Прочие замечания (не баги, зафиксировано)
- `grant_reward` currency использует `reason=EVENT_REWARD` ("reward"), которого
  нет в `PRODUCTIVE_REASONS` → ешки-награды из кейсов НЕ раздувают
  `total_earned` (как азартные игры). Сознательно консистентно с ECONOMY.md;
  если захотим, чтобы кейс-награды считались доходом — добавить reason в
  PRODUCTIVE_REASONS. Сейчас НЕ меняем.
- `reward_kind` БД допускает `tg_gift`/`stars`, но `open_case` фильтрует по
  `REWARD_KINDS_V1`. Админ-API теперь тоже отклоняет их при добавлении дропа —
  раньше можно было завести строку, которую рантайм молча игнорирует, искажая
  отображаемые шансы. Закрыто на уровне API.
- `case_openings.reward_id` — логическая ссылка без FK; при удалении строки
  дроп-листа历史 сохраняет `weight_snapshot`, `reward_id` может «повиснуть» —
  ожидаемо (история неизменна).

## 7. Риски

- **0016 не обкатана**: миграция/модели не прогонялись (нет локального Python —
  только Docker). До мержа обязателен прогон upgrade/downgrade + импорт
  приложения в Docker.
- **Атомарность open_case протестирована только чтением** + юнит на веса. Нужен
  интеграционный тест на PostgreSQL (FOR UPDATE не воспроизводится на sqlite).
- **Каталог item'ов**: Cases V1 предполагает, что предметы-кейсы и
  предметы-награды уже есть в `inventory_items`. Механизма «создать item из
  админки» в V1 нет — заводятся вручную/скриптом. Это вне скоупа кейсов, но
  блокирует наполнение реальными кейсами.
- **TS-линт «Cannot find module»** в редакторе — артефакт окружения (node_modules
  не установлены в песочнице), не ошибка кода; новые роуты повторяют рабочие.
