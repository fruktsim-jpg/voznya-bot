# MINI APP FOUNDATION — архитектура (без реализации)

Статус: **проект (design only).** Реализации нет. Этот документ описывает полную
архитектуру Telegram Mini App поверх существующего ядра: авторизация, профили,
инвентарь, кейсы, открытия, история, Gifts, Stars.

Дополняет `MINI_APP_PLAN.md` (там — детальный auth-flow, оценка строк/дней,
BotFather). Здесь — **архитектура фич и backend-контракт**, чтобы было видно, как
всё слоится поверх Cases V1 и существующих read-only API.

Дата: 2026-06-06
Связанные: `MINI_APP_PLAN.md`, `CASES_V1_PLAN.md`, `CASES_V1_READINESS.md`,
`CONTINUATION_REPORT.md` (§3 архитектурные решения), `FOUNDATION_STATUS.md`.

---

## 0. Незыблемые решения (контекст, не пересматриваем)

- **Бот = игровое ядро** (единственный писатель gameplay-таблиц). PostgreSQL —
  источник истины.
- **`open_case()` / `grant_reward()` — единственные реализации** логики открытия
  и выдачи. Никакой логики открытия в TypeScript. Mini App — клиент.
- **FastAPI пока НЕ внедряется.** До активной разработки Mini App открытие
  доступно только в боте (in-process). HTTP-слой ядра вводится ИМЕННО на этом
  этапе (см. §3) и только тогда.
- Сайт и Mini App переиспользуют одни и те же React-компоненты и один и тот же
  backend. Различие — только в авторизации (сессия сайта vs Telegram initData).

## 1. Архитектура (слои)

```
┌── Telegram client (WebView) ──────────────────────────────────────────────┐
│  Mini App (Next.js /app)  — React UI, переиспускает компоненты сайта        │
│     │  Authorization: Bearer <jwt>  (см. MINI_APP_PLAN §8)                   │
│     ▼                                                                       │
│  Next.js API (тонкий)                                                       │
│   • READ  /api/* — читают игровые таблицы напрямую (как сайт, read-only)    │
│   • WRITE gameplay — НЕ здесь. Проксируются в Python-ядро (open-endpoint)    │
└─────────────────────────────┬───────────────────────────────────────────────┘
                              │ (только для мутаций gameplay, напр. открытие кейса)
                              ▼
        Python HTTP-слой ядра (FastAPI в контейнере бота) — НОВОЕ на этом этапе
                              │  единственный путь мутаций: open_case/grant_reward
                              ▼
                        PostgreSQL (источник истины)
```

Принцип разделения:
- **Чтение** (профиль, инвентарь, витрина, история) — Next.js читает БД
  напрямую, как уже делает сайт (`lib/queries.ts`, `lib/cases.ts`). Дёшево,
  быстро, не требует Python-слоя.
- **Запись gameplay** (открыть кейс, позже — купить, подарить) — ТОЛЬКО через
  Python-ядро. Next.js не пишет gameplay (инвариант AGENTS.md). Для этого и
  вводится тонкий FastAPI в контейнере бота.

## 2. Авторизация

Полностью описана в `MINI_APP_PLAN.md` §2–§8. Кратко:
- `Telegram.WebApp.initData` → `POST /api/auth/telegram-webapp` →
  `verifyWebAppInitData` (HMAC по `BOT_TOKEN`, timing-safe, свежесть
  `auth_date`) → `createSessionToken({uid})` → Bearer-JWT.
- `initData.user.id` — это УЖЕ настоящий `user_id`, поэтому привязка через бота
  для входа НЕ нужна (в отличие от OIDC).
- `getSession()` дорабатывается: сначала `Authorization: Bearer`, потом cookie.
- **Для open-endpoint Python-ядра** Mini App шлёт тот же initData (или
  выданный сайтом краткоживущий токен), а FastAPI повторно валидирует подпись —
  ядро не доверяет «честному слову» Next.js, проверяет Telegram-подпись само.

Мост к аккаунту сайта (OIDC `account_links`) Mini App-входу не нужен, но
остаётся доступен (один и тот же `user_id`).

## 3. Python HTTP-слой ядра (вводится здесь, не раньше)

Тонкий FastAPI в том же контейнере/процессе, что и бот — общий доступ к
моделям, `open_case`, `grant_reward`, сессии БД и `DbSessionMiddleware`-аналогу.

Эндпоинты (минимум для кейсов):
| Метод | Путь | Действие | Внутри |
|-------|------|----------|--------|
| `GET`  | `/api/cases` | витрина активных кейсов | read (как `lib/cases.ts`) |
| `GET`  | `/api/cases/{code}` | содержимое + шансы | read |
| `POST` | `/api/cases/{code}/open` | **единственный путь открытия** | `open_case(session, user_id, code)` |
| `GET`  | `/api/cases/openings?user=` | история открытий | read `case_openings` |

Контракт open-endpoint:
- `user_id` берётся ИЗ авторизации (initData/токен), НЕ из тела запроса;
- внутри — ровно `open_case()` (одна транзакция, pre-flight, леджер);
- ответ = `OpenResult` (что выпало, новый баланс/кол-во) для анимации;
- идемпотентность/гонки — уже обеспечены блокировками в `open_case`.

Это единственная новая серверная логика этапа Mini App. Всё остальное —
переиспользование.

## 4. Профили

- Чтение карточки игрока — уже есть на сайте (`getPlayerProfile`,
  `getUserSummary` в `lib/queries.ts`, роуты `/api/me`, `/api/profile/[id]`).
- Mini App переиспользует те же запросы и компоненты `components/profile/*`.
- Источник — `users` (read-only). Mini App ничего в `users` не пишет.
- Экран: имя, титул, баланс, ранг, MMR, репутация, ачивки, серия — как
  `PROFILE` в боте/на сайте.

---

## 5. Инвентарь

- **V1 (стековый):** чтение `inventory` + каталог `inventory_items` (как
  `/inventory` в боте и инвентарь на сайте). Группировка по типу/редкости,
  визуал — `lib/inventory.ts` (`rarityStyle`, `typeEmoji`).
- **Позже (per-instance):** когда включат `inventory_instances` (Gifts/
  серийники), Mini App добавит просмотр уникальных экземпляров (серийник,
  «#3 из 100», состояние). Схема готова (миграция 0016), рантайма нет.
- Экипировка косметики (`inventory.equipped` + слот) — отдельный этап
  (Cosmetics), не часть фундамента.

---

## 6. Кейсы (витрина)

- Витрина активных кейсов — уже есть данные (`lib/cases.ts` →
  `getActiveCasesWithRewards`). Mini App рендерит те же карточки, что
  публичная `/cases`, в WebView-вёрстке.
- Содержимое + честные шансы (`weight/Σweight`) — read-only.
- Кнопка «Открыть» ведёт в open-flow (§3/§7), не реализует логику сама.

## 7. Открытия

- Нажатие «Открыть» → `POST /api/cases/{code}/open` (Python-ядро) → `open_case`.
- Ответ → клиентская анимация (рулетка/вскрытие), подсветка редкости из
  `RARITY_STYLES`, нативный `HapticFeedback`/`MainButton` Telegram.
- Никакой предрешённости на клиенте: результат приходит с сервера (CSPRNG в
  `_pick_reward`), анимация лишь «доигрывает» к уже известному исходу — честно
  и неподделываемо.
- Состояния ошибок (нет ключа, не хватает ешек, неактивен) — из `OpenResult`.

## 8. История

- Чтение `case_openings` (свои открытия) — `GET /api/cases/openings?user=`.
- Каждая строка проверяема: `roll` + `weight_snapshot` (provably-fair-ready).
- Экран «Мои открытия»: что выпало, когда, из какого кейса; разворот снимка
  весов для спорных случаев.
- Дополнительно: `inventory_history` (движение предметов) и `transactions`
  (движение ешек) — уже читаемы, дают полную «выписку».

## 9. Gifts (Telegram Gifts) — пост-фундамент

- Строятся поверх `inventory_instances` (own/pending/granted/failed,
  `telegram_gift_id`, серийники) — схема готова, рантайма нет.
- Новая ветка в `grant_reward` (`tg_gift`): создать
  `inventory_instances(state=pending)` + задача на `sendGift` → по успеху
  `granted`, по ошибке `failed`. Это и есть «точка невозврата» (первая выдача
  уникального предмета) — до неё всё обратимо.
- Mini App: экран подарков (свои/входящие), отправка подарка (через ядро),
  отображение серийников. Только UI + вызовы ядра; логика — в Python.

## 10. Stars (Telegram Stars) — пост-фундамент

- Платное открытие/покупки: `case_definitions.open_cost_kind='stars'` (задел
  есть) и/или `reward_kind='stars'`.
- Поток оплаты — нативный Telegram Stars (invoice в WebApp) → webhook
  подтверждения в Python-ядро → только после подтверждения вызывается
  `open_case`/выдача. Деньги Stars не смешиваются с ешками (это разные валюты).
- Требует: интеграция `telegram_payments`, проверка платежа на стороне ядра,
  идемпотентность по payment id. Снятие запрета `open_cost_kind='stars'` в
  админ-валидации (сейчас намеренно заблокировано).

## 11. Что переиспользуется vs что новое

| Слой | Источник | Статус |
|------|----------|--------|
| Auth (initData→JWT, Bearer) | `MINI_APP_PLAN.md` | спроектировано, ~готово на 80% |
| Read API (профиль/инвентарь/витрина/история) | `lib/queries.ts`, `lib/cases.ts` | **есть** (read-only) |
| Open-flow | `open_case`/`grant_reward` | **есть** (ядро); нужен HTTP-обвяз |
| Python HTTP-слой (FastAPI) | — | **новое**, минимальный (cases-эндпоинты + auth-мост) |
| UI-компоненты | `components/*` | **есть** (переиспускаем) |
| TMA-обвязка (SDK, хук, провайдер, gate, layout) | — | **новое** (~9 файлов, `MINI_APP_PLAN.md` §4) |
| Gifts / Stars | — | **новое**, пост-фундамент (отдельные этапы) |

## 12. Порядок (когда дадут добро)

1. **Фундамент** (`MINI_APP_PLAN.md` §10): auth-роут + Bearer в `getSession` +
   TMA-обвязка + layout/экран + menu button в боте + BotFather.
2. **Python HTTP-слой ядра** (§3): FastAPI с `/api/cases` + `/open` + auth-мост.
   Это первый ввод HTTP-API ядра — раньше не вводим.
3. **Кейсы в Mini App**: витрина → открытие через open-endpoint → анимация →
   история.
4. **Cosmetics** (экипировка) — отдельно.
5. **Gifts** (поверх `inventory_instances`) — отдельный этап, точка невозврата.
6. **Stars** (платежи) — отдельный этап.
