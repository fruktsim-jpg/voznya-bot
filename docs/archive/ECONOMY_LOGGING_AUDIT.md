# Economy Logging Audit & Plan

Аудит того, какие экономические события сейчас оставляют след в БД, чего не
хватает для полноценной аналитики, и какие точки логирования нужно заложить
**сейчас**, чтобы через полгода можно было построить любую статистику по истории
проекта без догадок и ручных пересчётов.

> Только аудит, рекомендации и план. Новые системы здесь не реализуются.
> Принцип: если есть сомнение — логировать.

Дата: 2026-06-07. HEAD миграций: `0018_gift_catalog`.

---

## 0. TL;DR

- Денежный леджер (`transactions`) — единственный надёжный источник правды по
  ешкам, и он работает: каждое движение баланса = одна строка.
- **Хорошая новость:** инфраструктура журналов уже заложена в foundation-миграциях
  (`purchase_history`, `gift_transactions`, `case_openings` — у всех есть
  `transaction_id`, `audit_id`, `meta`). Их почти не нужно расширять — нужно
  начать **писать** в них из рантайма.
- **Дыры, которые нельзя восстановить задним числом** (если не начать сейчас):
  Telegram Stars / донат, покупки Gifts, фонд Gifts, связь открытия кейса с
  его транзакцией, эталонная стоимость предметов (для EV и P&L).
- Рекомендация: 1 маленькая правка кода (`change_balance` → вернуть транзакцию),
  1 узкая миграция (стоимость предмета), и дисциплина «писать в уже существующие
  журналы» при запуске каждой новой механики.

## 1. Что уже логируется (есть след)

### 1.1. `transactions` — денежный леджер (РАБОТАЕТ)
`change_balance()` (`app/services/economy.py`) — единственный мутатор баланса.
Каждый вызов пишет `Transaction(user_id, amount, reason, meta, created_at)`.
14 точек вызова, все с осмысленным `meta`:

| Источник | reason | meta-ключи |
|---|---|---|
| Ферма | `farm` | `outcome, base, streak` (только при amount≠0) |
| Казино | `casino` | `bet, multiplier, payout, outcome` |
| Дуэль (ставка ×2) | `duel` | `role=stake` |
| Дуэль (выигрыш) | `duel` | `role=win, bank` |
| Клад | `treasure` | `treasure_id` |
| Ачивка | `achievement` | `code` |
| Номинация (пидор) | `nomination` | `type=pidor` |
| Номинация (пара) | `nomination` | `type=para` |
| Админ выдал | `admin` | `action=give` |
| Админ снял | `admin` | `action=take` |
| Покупка кейса | `purchase` | `source=case_open, case` |
| Награда кейса (ешки) | `reward` | `source, …` |

Аналитика, доступная **сегодня** из одного `transactions`: вся денежная масса,
эмиссия/сток по дням и по источникам, активные игроки, средний баланс, экономика
казино (RTP/прибыль/топы), валютная часть кейсов. Это и питает Economic Control
Center.

### 1.2. Прочие журналы (РАБОТАЮТ)
- `case_openings` — каждое открытие кейса: `reward_kind, reward_item_code,
  amount, qty, roll, weight_snapshot, created_at`. Провабли-фейр снапшот весов
  есть → распределение наград и фактические шансы считаются точно.
- `reputation_entries`, `mmr_entries` — полная история репутации и MMR с причинами.
- `message_daily` — дневные счётчики сообщений (для сайта).
- `audit_log` — все админ-действия (кто/над кем/почему).
- `user_achievements` — выданные ачивки.

### 1.3. Журналы, которые СУЩЕСТВУЮТ, но в них НИКТО не пишет (foundation-only)
Это ключевой вывод аудита — каркас уже есть, не хватает рантайм-писателей:

- **`purchase_history`** (миграция 0010): `user_id, offer_id, item_code, price,
  quantity, source, transaction_id, audit_id, meta, created_at`. Готовый журнал
  покупок в магазине. Пустой — потока покупок нет.
- **`shop_offers`** (0010): товары с `price, sold_count, max_supply,
  per_user_limit, сезонность`. Пустой.
- **`gift_transactions`** (0011): `kind, gift_type, sender_user_id,
  recipient_user_id, item_code, quantity, amount, status, idempotency_key,
  transaction_id, audit_id, meta`. Журнал **передач между игроками** (player↔player
  дарение предметов/ешек). НЕ путать с покупкой реального Telegram Gift из
  `gift_catalog`. Пустой.
- **`gift_catalog`** (0018): каталог Telegram Gifts (`star_cost, price_eshki,
  sold_count, stock, reserved`). Заполняется через админку, но `sold_count`
  всегда 0 — продаж нет.

## 2. Чего НЕ хватает (нельзя посчитать сейчас)

### 2.1. Запрошенные точки

| # | Что | Статус сейчас | Почему важно |
|---|---|---|---|
| A | `transaction_id` в `case_openings` | колонка есть, всегда NULL | связать открытие с его списанием (точный per-open burn, аудит, дедуп) |
| B | Учёт покупок Gifts | нет следа | сколько подарков куплено, кем, за сколько ешек |
| C | Учёт фонда Gifts | нет сущности | приход ешек игроков vs расход Stars владельца = баланс фонда |
| D | Реальный P&L по Gifts | невозможен | нужна выручка (B) и себестоимость в Stars (расход) |
| E | Полный EV кейсов | только валютная часть | предметные награды без стоимости → EV занижен/неполный |

### 2.2. Корневые причины
- **A:** `change_balance()` возвращает `User`, а не `Transaction`. В `open_case()`
  результат списания вообще отбрасывается (`await change_balance(...)` без
  присваивания). Поэтому id транзакции некуда взять.
- **B:** потока покупки Gifts не существует (нет хендлера/роут, нет записи в
  `purchase_history`/`gift_catalog.sold_count`).
- **C:** нет ни таблицы фонда, ни записи расхода Stars. Stars-расход владельца
  нигде не фиксируется.
- **D:** следствие B+C.
- **E:** у `inventory_items` нет поля «эталонная стоимость в ешках». Награда-предмет
  в `case_openings` есть, но в деньгах её не оценить.

### 2.3. События, которые НЕВОЗМОЖНО восстановить задним числом
Это самое критичное — если не начать писать сейчас, история потеряна навсегда:

1. **Telegram Stars / донат.** Сейчас в `app/` нет ни одного payment-хендлера
   (`successful_payment`, `pre_checkout_query`), нет таблицы платежей. Любой
   донат, прошедший до внедрения логирования, в аналитике не появится никогда.
2. **Расход Stars владельца на отправку Gifts.** Если отправлять подарки без
   записи себестоимости в момент отправки — P&L по Gifts восстановить нельзя.
3. **Курс/наценка на момент сделки.** `1 Star ≈ 10 ешек` и наценка будут меняться.
   Если не фиксировать цену сделки в момент покупки (price snapshot), пересчёт
   исторического P&L станет догадкой.
4. **Покупки Gifts/магазина без записи в `purchase_history`.** `sold_count` —
   агрегат, он не хранит кто/когда/почём.

> Денежная часть (ешки) задним числом восстановима из `transactions` всегда.
> Невосстановимо именно то, что лежит **вне** леджера ешек: Stars, себестоимость,
> внешние платежи, контекст сделки.

## 3. План: изменения в БД

Минимум новых сущностей. Сначала — задействовать существующее.

### 3.1. Изменения, нужные уже сейчас (низкий риск)

**(M1) Эталонная стоимость предмета — для EV и P&L.**
Узкая миграция: добавить в `inventory_items` колонку
`ref_value BIGINT NULL` (оценочная стоимость предмета в ешках; NULL = не оценён).
Не трогает экономическое ядро, только справочное поле каталога.
→ Разблокирует полный EV кейсов (E) и оценку предметных наград/подарков.

**(M2) Никаких новых таблиц для A.** `case_openings.transaction_id` уже есть —
нужна только правка кода (см. §4).

### 3.2. Что использовать как есть (без миграций)
- **Покупки Gifts (B)** → писать в существующий `purchase_history` с
  `source='gift'`, `item_code=<gift code>`, `transaction_id` от списания ешек,
  `meta={star_cost, price_eshki_snapshot, fx_rate}`. Таблица уже под это
  заточена (`source`, `transaction_id`, `meta`).
- **Передачи между игроками** → существующий `gift_transactions`.

### 3.3. Новые сущности — только там, где действительно нет каркаса

**(M3) Учёт Stars / донат / фонд — `stars_ledger` (один журнал, отложенная реализация).**
Сейчас Stars нигде не живут. Рекомендуемая единая таблица движения Stars
(аналог `transactions`, но в Stars), чтобы и донат, и расход на Gifts, и баланс
фонда считались из одного места:

```
stars_ledger
  id            BIGINT PK
  user_id       BIGINT NULL      -- кто (донатер) / NULL для системных
  amount        BIGINT           -- знаковый: + приход Stars, − расход
  reason        VARCHAR(32)      -- 'donate' | 'gift_send' | 'refund' | 'adjust'
  ref_type      VARCHAR(32) NULL -- 'gift_catalog' | 'purchase' | ...
  ref_id        VARCHAR(64) NULL -- ссылка на связанную сущность
  telegram_charge_id VARCHAR(128) NULL  -- id платежа Telegram (дедуп!)
  meta          JSONB NULL
  created_at    TIMESTAMPTZ
```

Баланс фонда Gifts (C) = `SUM(amount)` по `stars_ledger`. P&L Gifts (D) =
выручка в ешках (из `purchase_history`) против расхода Stars (из `stars_ledger`).
**Эту таблицу стоит создать в момент, когда реально появится донат/отправка
Gifts** — но спроектировать и согласовать схему сейчас, чтобы `telegram_charge_id`
писался с самого первого платежа (иначе дедуп и сверка с Telegram невозможны).

## 4. План: изменения в логировании (код)

Все правки — точечные, поверх существующих единых точек. Реализация — отдельной
задачей; здесь фиксируем что и где.

**(L1) `change_balance` → возвращать транзакцию.**
Сейчас возвращает `User`. Варианты без ломки 14 вызовов:
- добавить `await session.flush()` после `session.add(Transaction(...))` и
  возвращать кортеж/объект, **либо** ввести тонкий хелпер
  `change_balance_tx(...) -> Transaction`, который вызывают только новые места
  (кейсы, магазин, Gifts). Предпочтительно второе — нулевой риск для старого кода.

**(L2) `open_case()` → проставлять `case_openings.transaction_id`.**
Захватить транзакцию списания (L1) и передать её id в конструктор `CaseOpening`.
Также заполнять `audit_id`, если открытие инициировано админом.
→ Закрывает (A): точный per-open burn, провабли-связь открытие↔деньги.

**(L3) Поток покупки Gifts (при запуске магазина).**
В момент покупки в одной транзакции: списать ешки (`reason='purchase'`,
`meta.source='gift_buy'`), записать `purchase_history(source='gift',
transaction_id=…, meta={star_cost, price_eshki, fx_rate})`, инкремент
`gift_catalog.sold_count`. → Закрывает (B) и даёт выручку для (D).

**(L4) Отправка Gifts → расход Stars.**
В момент фактической отправки подарка писать `stars_ledger(reason='gift_send',
amount=-star_cost, ref_type='gift_catalog', ref_id=<code>,
telegram_charge_id=…)`. → Закрывает расходную часть (C)/(D).

**(L5) Платёжные хендлеры Telegram (донат Stars).**
Добавить обработчики `pre_checkout_query` и `successful_payment`; на каждый
успешный платёж — строка `stars_ledger(reason='donate', amount=+stars,
telegram_charge_id=…, user_id=…)`. **Это самый невосстановимый кусок — заложить
первым, как только включается приём Stars.**

**(L6) Гигиена `meta` (дёшево, делаем заодно).**
Договориться о стабильном ключе `source` в `meta` для всех «покупочных»
списаний (`case_open`, `gift_buy`, `shop_buy`), чтобы аналитика группировалась по
`meta->>'source'`, а не по эвристикам. Уже частично есть у кейсов.

## 6. Аудит «что начать сохранять сейчас» по будущим системам

Принцип: любая важная операция должна оставлять след. Ниже — по каждой будущей
системе: что уже есть, чего не хватает, что невосстановимо.

### 6.1. Telegram Gifts
- Есть: каталог (`gift_catalog`).
- Не хватает: покупки (`purchase_history` source='gift'), расход Stars
  (`stars_ledger`), snapshot цены/курса в meta.
- Невосстановимо: себестоимость в Stars и `telegram_charge_id` отправки, если не
  писать в момент отправки.

### 6.2. Донат Stars
- Есть: ничего.
- Не хватает: payment-хендлеры + `stars_ledger(reason='donate')`.
- Невосстановимо: **всё**. Платёж, не записанный в момент `successful_payment`, в
  истории не появится. Обязательно хранить `telegram_charge_id` (дедуп и сверка).

### 6.3. Магазин (ешки → предметы)
- Есть: полный каркас (`shop_offers`, `purchase_history`).
- Не хватает: рантайм-писатель при покупке (списание + `purchase_history` +
  `sold_count`), `transaction_id` в записи.
- Невосстановимо: контекст сделки (цена/лимит на момент покупки), если не писать
  снапшот в `purchase_history.price` + `meta`.

### 6.4. Кейсы
- Есть: `case_openings` с провабли-снапшотом весов.
- Не хватает: `transaction_id` (L2), `ref_value` предметов для EV (M1).
- Невосстановимо: связь открытие↔транзакция за прошлый период (но открытия и так
  лежат, деньги — в леджере; связь восстановима приблизительно по времени).

### 6.5. Новые казино-режимы
- Есть: модель «1 спин = 1 транзакция `casino` с `meta{bet,payout,...}`».
- Рекомендация: сохранить этот контракт и для новых режимов; добавить в `meta`
  ключ `game` (например `slots`/`roulette`/`dice`), чтобы разрезать аналитику по
  режимам. Без этого режимы сольются в один «casino».
- Невосстановимо: разбивка по режимам за период до появления `meta.game`.

### 6.6. Mini App
- Есть: ничего специфичного.
- Рекомендация: когда появится, любое действие через Mini App, меняющее
  экономику, должно идти через те же единые точки (`change_balance`,
  `purchase_history`, `stars_ledger`) с `meta.channel='miniapp'`, чтобы можно
  было сравнивать каналы (бот vs Mini App vs сайт). Заложить ключ `channel` в
  `meta` для всех экономических операций сразу — дёшево и невосстановимо потом.

### 6.7. Экономика в целом
- Единый принцип: **один источник правды на каждый тип актива** —
  `transactions` (ешки), `stars_ledger` (Stars). Всё остальное (purchase_history,
  gift_transactions, case_openings) — журналы-сателлиты со ссылкой
  `transaction_id`/`ref_id` на источник правды.
- Сквозные ключи `meta`, которые стоит писать везде с самого начала:
  `source` (что за операция), `channel` (bot/site/miniapp), `game` (для казино),
  ценовые снапшоты для сделок.

## 7. Порядок внедрения (рекомендация)

Делать отдельными задачами, по возрастанию срочности «невосстановимости»:

1. **L1+L2** (transaction_id у кейсов) — маленько, чистый выигрыш, разблокирует
   точный per-open P&L. Низкий риск.
2. **M1** (`inventory_items.ref_value`) — узкая миграция, разблокирует полный EV.
3. **Согласовать схему `stars_ledger` (M3)** на бумаге сейчас, создать в момент
   запуска доната/Gifts.
4. **L3** (покупки Gifts в `purchase_history`) — при запуске Gifts-магазина.
5. **L4+L5** (Stars: отправка Gifts + донат) — одновременно с приёмом Stars; это
   самый невосстановимый слой, его нельзя откладывать после включения платежей.
6. **L6 + meta-гигиена** (`source`/`channel`/`game`) — внедрять в каждую новую
   механику с первого коммита.

> Экономическое ядро (`change_balance`, `transactions`), OIDC/`account_links` и
> прошлые миграции не трогаем без явной необходимости и тестов (AGENTS.md).
