# Admin V2 — финальная ревизия (что управляется через сайт, что требует деплоя)

Дата: финальная фаза Admin V2. Цель — зафиксировать, какие параметры систем
уже управляются из админки (`v0-voznya/app/admin/*`), а какие пока требуют
деплоя/миграции. Принцип ревизии: **не плодить настройки ради красоты**. Если
параметр уже живёт в БД (своя таблица) — управляем через профильный редактор;
если захардкожен в коде — переносим в `app_settings` только когда реально
понадобится менять.

---

## Кейсы

Источник истины: таблицы `case_definitions` + `case_rewards` (БД).

| Параметр | Где управляется | Деплой? |
|---|---|---|
| Существование/активность кейса | `/admin/cases` (POST `/api/admin/cases`) | нет |
| Стоимость открытия (kind/amount) | `/admin/cases` | нет |
| Расходует ли ключ (consumes_key) | `/admin/cases` | нет |
| Награды, веса, джекпот | `/admin/cases/[code]/rewards` | нет |
| Аналитика: открытия, RTP, premium/limited/jackpot, biggest/latest | `/admin/cases` (read-only) | — |

Вывод: **кейсы полностью управляются через сайт.** Правки наград не переписывают
историю — меняются только будущие шансы. Захардкоженного в коде, что требовало бы
переноса, не осталось (механика ролла — это код, не «настройка»).

---

## Магазин (Telegram Gifts)

Источник истины: таблица `gift_catalog` (БД).

| Параметр | Где управляется | Деплой? |
|---|---|---|
| Позиция каталога (имя, описание, цена ешки, star_cost, telegram_gift_id) | `/admin/gifts` (POST `/api/admin/gifts`) | нет |
| Запас/резерв/sort_order/активность | `/admin/gifts` | нет |
| Аналитика жизненного цикла предмета (куплено/из кейсов/выведено/другу/в инвентарях/возвраты/продано/выручка) | `/admin/shop` (read-only) | — |
| P&L подарков, фонд Stars | `/admin/economy/gifts` (read-only) | — |

Вывод: **магазин полностью управляется через сайт.** Цены и состав меняются без
деплоя. `is_limited` берётся из `inventory_items` (каталожный флаг) — это
свойство предмета, а не настройка магазина.

## Казино

Источник истины: код `app/settings/balance.py`, переопределяемый через
`app_settings` (`app/settings/dynamic.py`).

| Параметр | Где управляется | Деплой? |
|---|---|---|
| `casino.min_bet` | `/admin/settings` (app_settings) → бот читает через `dynamic.get_int` | нет (TTL кэша ~60с) |
| `casino.max_bet` | `/admin/settings` | нет |
| Таблицы выплат/множители/шансы | код `casino/handlers.py` | **да** |
| Аналитика: ставки, RTP, прибыль казино, крупнейшие win/loss | `/admin/economy/casino` (read-only) | — |

Вывод: лимиты ставок управляются с сайта. Математика выплат остаётся в коде —
**переносить не нужно**, пока нет реальной потребности менять её на лету (по
принципу «не мигрируем ради красоты»). Когда понадобится — ключи кладутся в
`app_settings` тем же паттерном, что и `casino.min_bet`.

## Доставки

Источник истины: таблица `gift_transactions` (kind='tg_gift').

| Действие | Где управляется | Деплой? |
|---|---|---|
| Просмотр очереди/ошибок/выданных/возвратов + статистика за период | `/admin/deliveries` (read-only) | — |
| Выдать вручную (complete) | `/admin/deliveries` (POST) | нет |
| Возврат (refund) | `/admin/deliveries` | нет |
| Повторить авто-выдачу (retry → бот sendGift) | `/admin/deliveries` | нет* |
| Фильтры: статус/источник/период, поиск | `/admin/deliveries` | — |

\* retry требует настроенных `BOT_INTERNAL_URL` / `BOT_INTERNAL_SECRET`. Если
внутренний API не сконфигурирован — retry вернёт 503, но фоновый воркер бота
повторит сам, а complete/refund доступны всегда.

Вывод: **доставки полностью управляются через сайт.** Telegram-команды
`/gifts_pending`, `/gifts_done`, `/gifts_refund`, `/gifts_retry` дублируются на
100% (retry — при настроенном внутреннем API). «failed» — производное
представление pending+attempts (у бота нет отдельного статуса failed; это
осознанно, без миграции).

## Экономика

Источник истины: `transactions` (леджер) + `users.balance`.

| Раздел | Где | Деплой? |
|---|---|---|
| Сводка: всего/средний/медианный баланс, топ 1%, активные, эмиссия за день | `/admin/economy` (read-only) | — |
| Поток за 14 дней (создано/сожжено/нетто) | `/admin/economy` | — |
| Источники за 30 дней (по reason) | `/admin/economy` | — |
| Генерация/уничтожение по категориям (ферма/клад/кейсы/казино/админ/возвраты/подарки ↔ кейсы/магазин/казино/передачи) | `/admin/economy` | — |
| Распределение богатства (0–100 / 100–1000 / 1000–10000 / 10000+) | `/admin/economy` | — |
| Богатейшие игроки | `/admin/economy` | — |
| Ручные начисления/списания игроку | `/admin/players/[id]` | нет |

Вывод: **аналитика экономики полностью на сайте.** Категории сгруппированы из
`reason`/`meta.source` — если в боте появится новый источник, достаточно
дополнить маппинг в `lib/economy-analytics.ts` (код сайта, без миграции).

## Аудит

Источник истины: таблица `audit_log`.

| Возможность | Где | Деплой? |
|---|---|---|
| Журнал: кто / когда / что / над чем | `/admin/audit` | — |
| Фильтры: user, action (prefix), даты | `/admin/audit` | — |
| Старое → новое значение | `/admin/audit` (из `meta.old`/`meta.new`) | — |

Изменения настроек теперь пишут в `audit_log.meta` пару `old`/`new`
(`settings.update`), и UI рисует диф `было → стало`. Остальные write-эндпоинты
(кейсы, подарки, доставки) уже писали ключевые поля в `meta`; формат `old/new`
можно добавлять им точечно по мере надобности тем же паттерном.

## Итог

| Система | Управление с сайта | Что ещё требует деплоя |
|---|---|---|
| Кейсы | ✅ полностью | механика ролла (код) |
| Магазин | ✅ полностью | — |
| Доставки | ✅ полностью | retry — при настроенном внутреннем API |
| Экономика | ✅ аналитика + ручные правки | новые источники → маппинг в коде |
| Казино | ✅ лимиты ставок | таблицы выплат (код, по потребности) |
| Аудит | ✅ полностью, с old→new | расширение old/new на прочие экшены — по потребности |

Инфраструктурный этап закрыт: ключевые решения принимаются через сайт, а не
через SQL/Docker/логи/Telegram-команды. Дальнейшие работы — в игровые механики.
