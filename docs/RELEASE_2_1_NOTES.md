# VOZNYA RELEASE 2.1 — Магазин, Инвентарь и Монетизация

Прогресс реализации по приоритетам ТЗ. Бэкенд-бот (этот репозиторий) и
мини-апп (`v0-voznya`) — разные кодовые базы; ниже разделено по слоям.

## Сделано (bot / backend)

### P0 — возврат кейсовых призов (КРИТИЧНО, деньги игрока)
`app/features/gifts/service.py` → `_refund`:
при отмене **приза кейса** теперь начисляется полная внутренняя стоимость
предмета (Вариант А): `star_cost × ESHKI_PER_STAR`. Источник `star_cost` —
живой каталог, затем слепок `meta` доставки. Запись эмиссии — через
экономическое ядро (`reason=reward`), помечается в `meta.refund_amount`.
- Курс: `ESHKI_PER_STAR = 10` в `app/settings/balance.py`.
- Тест: `tests/test_gift_refund_value.py` (контрольные примеры из ТЗ:
  Сердечко 150, Роза 250, Ракета 500, Бриллиант 1000, Premium 10000/15000).

### P5 — продажа предметов (70%)
`app/features/gifts/service.py` → `sell_gift` + `_item_full_value` +
`_sell_value` (`ITEM_SELL_RATE = 0.70`). Продаёт pending-предмет (приз кейса
или покупку магазина) за ешки, переводит доставку в `cancelled`, для покупок
освобождает резерв каталога. Атомарно и идемпотентно (FOR UPDATE строки
доставки), проверка владельца.
- Тест: `tests/test_gift_sell_value.py` (105/175/350/700/7000/10500).

### P7 / P1 (бот) — экран выбора после выпадения подарка
`app/features/cases/*` + `app/core/keyboards.py`:
после выпадения tg_gift из кейса показываем стоимость и кнопки
**«Оставить (доставка)» / «Продать за N»**. Колбэки `gift:keep` / `gift:sell`
в `app/features/cases/handlers.py` (защита владельца, снятие клавиатуры).

### P3 — Магазин вместо Подарков (бот)
`/магазин` и `/shop` — основные команды витрины (`/подарки`, `/gifts` остались
алиасами). Заголовок и help обновлены («🛒 Магазин … Telegram Gifts и Premium»).

## Осталось (mini-app `v0-voznya`, Next.js — отдельная задача)

Эти пункты — фронтенд мини-аппа и требуют отдельной реализации/сборки:

- **P1 (web)** — экран выбора после открытия кейса в мини-аппе (бот уже умеет).
- **P2** — полноценный раздел `/inventory` с карточками предметов, Premium
  отдельным блоком (активировать/продать/подарить).
- **P4** — кнопка «Вывести» из инвентаря → создание pending `GiftTransaction`
  (бэкенд-конвейер `deliver/refund/sell` уже готов и переиспользуется).
- **P6** — мгновенное обновление баланса (React Query invalidate) после
  открытия кейса / продажи / покупки / награды.

> ПОПРАВКА (см. `RELEASE_2_1_AIOGRAM_AND_WEB.md`): сайт НЕ ходит во внутренний
> API бота — он открывает кейсы напрямую через общую БД (`lib/cases-open.ts` —
> TS-порт `open_case`). Поэтому web-продажа/вывод делаются TS-логикой против
> той же БД (новые `/api/inventory/sell`, `/api/inventory/withdraw` по образцу
> `/api/cases/open`), а НЕ через `app/web/internal_api.py`. Серверные формулы
> уже есть в боте (`sell_gift`/`_sell_value`/`ITEM_SELL_RATE`) — порт обязан
> повторять их 1:1.

Также: апгрейд aiogram 3.13→3.28 потребовал поднять пин в `requirements.txt`
(иначе пересборка образа откатывала бы версию). Разбор последствий апгрейда и
ответы по авто-выдаче/Premium — в `docs/RELEASE_2_1_AIOGRAM_AND_WEB.md`.
