## 1. Спецификация вайпа (table-level)

### 🔴 Сбрасывается

| Сущность | Таблица / поле | Как |
|---|---|---|
| Баланс ешек | `users.balance` | → 0 |
| Прогресс заработка | `users.total_earned`, `users.total_spent` | → 0 |
| Счётчик успешных ферм | `users.farm_success_count` | → 0 |
| MMR (значение) | `users.mmr` | → 0 |
| MMR (журнал) | `mmr_entries` | TRUNCATE (после снапшота в архив) |
| Титулы | производное от `total_earned` | обнулится автоматически (earned=0) |
| Достижения + прогресс | `user_achievements` (+ прогресс-таблицы) | TRUNCATE |
| Экономический леджер | `transactions` | TRUNCATE (после архива) |
| Статистика кейсов | `case_openings` | TRUNCATE (после архива) |
| Статистика кладов | журнал кладов / `transactions reason=treasure` | TRUNCATE |
| Рейтинги/лидерборды | производные от выше | обнулятся сами |
| Инвентари | `inventory`, `inventory_history` | TRUNCATE |
| Лимитки / Premium / предметы | `gift_transactions` (kind='tg_gift'), резервы `gift_catalog` | TRUNCATE gt; `reserved=0, sold_count=0` |
| Сезонные награды | новые сезонные таблицы | пусто на старте |
| Аудит | `audit_log` | TRUNCATE (после снапшота в архив) |


### 🟢 НЕ сбрасывается

| Сущность | Таблица / поле |
|---|---|
| Аккаунт пользователя | `users` (строка остаётся) |
| Telegram ID | `users.user_id` |
| История сообщений / счётчик | `users.messages_count` (+ Combot overlay) |
| Исторические журналы/логи | архивные таблицы |
| Архивы прошлых сезонов | `season_archive_*` |
| Роли админов | `admin_roles` |

| Каталоги (определения) | `case_definitions`, `case_rewards`, `gift_catalog` (сами позиции, без счётчиков) |

> Важно: вайп НЕ трогает `users` как запись и `messages_count`. Старый аккаунт
> теряет всё игровое, но не теряет идентичность и историю чата. Преимущества
> старых аккаунтов на старте Сезона 1 — нет.

### Порядок операций вайпа (одна транзакция-обёртка где возможно)

1. Снапшот в архив: `transactions`, `mmr_entries`, текущие `users.balance/mmr/
   total_earned`, лидерборды, инвентари → `season_archive_*` (zero-loss истории).
2. TRUNCATE игровых журналов: `transactions`, `mmr_entries`, `case_openings`,
   `inventory`, `inventory_history`, `user_achievements`, `gift_transactions`.
3. UPDATE `users SET balance=0, total_earned=0, total_spent=0,
   farm_success_count=0, mmr=0`.
4. UPDATE `gift_catalog SET reserved=0, sold_count=0`.
5. Сид `season` (Сезон 1, 56 дней) + сезонный контент.

## 2. Экономика — точные расчёты дохода

Базовые EV из конфига: ферма EV ≈ 6.26 (до 7.57 серией +20%), кд 4ч → **6 ферм/
день**; клад ср. 35, 2–4 спавна/день; номинации ≤ +4/день; казино EV 0.98 (не
доход); дуэли zero-sum.

### Доход активного честного игрока

| Период | Ферма | Клад (реалистично) | Номинации | Итого «честно» |
|---|---|---|---|---|
| День | 6×6.26≈**38** (до 45 серией) | ~10 (не каждый спавн его) | ~2 | **~50/день** |
| Неделя | ~266–315 | ~70 | ~14 | **~350/нед** |
| Сезон (56 дн) | ~2 130–2 520 | ~560 | ~110 | **~2 800–3 200/сезон** |

С новым daily/weekly/mission-слоем (см. §6) добавляется управляемый поток —
закладываем его так, чтобы **сезонный потолок честной игры ≈ 5 000–6 000 ешек**.
Этого хватает на 💎 Джекпот-кейс пару раз, но не на «всё сразу».

### Максимум активного игрока (хардкор, всё по максимуму)

| Источник | Макс/день |
|---|---|
| Ферма (6× с серией +20%) | ~45 |
| Daily reward (см. §6) | ~30 |
| Weekly mission (амортизировано) | ~30 |
| Клад (если быстрый) | ~35 |
| Номинации | ~4 |
| **Итого потолок** | **~145/день → ~8 000/сезон** |

Достижения (разово, ~6 585 за всё время) — отдельный единоразовый буст, размазан.

### Максимум через абуз (ДО фиксов — что закрываем)

| Вектор абуза | Потенциал | Фикс |
|---|---|---|
| Мультиаккаунты на ферме | N×45/день линейно | §8: порог активности + лимит |
| Перелив ешек на главный акк | весь доход доноров | §8: дневной лимит передач |
| Дуэль-фарм MMR (2 акка) | +25 MMR/бой ∞ | §8: лимит участий/день, разные оппоненты |
| Казино-принтер | устранён ранее (EV 0.98) | 🟢 |

После фиксов §8 абузный потолок сводится к честному ×(число живых аккаунтов с
реальной активностью), а перелив капается дневным лимитом.

### Прогноз инфляции на сезон

Sink (кейсы RTP<1, магазин, продажа −30%, казино −2%) при честном потоке ~50–145/
день **превышает** mint у активных тратящих игроков. Накопители (не тратят) —
основной источник массы; их потолок ~8 000/сезон по дизайну. Вывод: **повторный
вайп через 2 месяца не нужен**, если §8 закрыт и daily/weekly не раздуты.

## 3. Кейсы — ревизия

Существующие 6 кейсов (RTP 0.826–0.962) — **🟢 оставить** (см. таблицу V2).

Добавить к Сезону 1:

| Новое | Решение |
|---|---|
| Сезонный кейс «🏆 Сезон 1» | цена ~600, RTP ~0.88, пул = сезонные лимитки |
| Сезонные лимитки | отдельные `item_code` с `season_id=1`, выпадают только из сезонного кейса |
| Новые типы наград | **сезонные очки** (валюта рейтинга, не выводится) как редкая награда — опционально 🟡 |

RTP сезонного кейса держим <1 (сток). Сезонные лимитки **не продаются обратно**
за полную цену (sell-rate ниже или 0), чтобы не стали ешко-краном.

## 4. Ферма — ревизия (ежедневная мотивация)

Текущее: 5 исходов (common/uncommon/rare/zero/loss), серия-бонусы 3/7/14/30 дней.

| Изменение | Решение | Приоритет |
|---|---|---|
| Доходность | EV ~6.26 оставить (сбалансирован) | 🟢 |
| Редкости фермы | оставить 5 исходов | 🟢 |
| События фермы | редкое «золотое зерно» (×3 награда, ~1%) для яркости | 🟡 |
| Апгрейды фермы | уровни за ешки (ешко-сток + цель) | 🟡 Сезон 2 |
| Мотивация заходить ежедневно | серия-бонусы УЖЕ есть; усилить связкой с daily (§6) | 🔴 |

Связать ферму с daily-стриком: пропуск дня сбрасывает серию — это и есть
ежедневный крючок без раздувания EV.

## 5. MMR — ревизия

Текущий формат (ферма+клад+дуэль+ачивки, сообщения не дают) — **🟢 оставить как
базу**, но:

| Вопрос | Решение |
|---|---|
| Нужен ли текущий формат | да — он не завязан на деньги, это честный прогресс |
| Отвязать от части активностей | **отвязать дуэль-участие** от безусловного +25 (анти-фарм, §8) |
| Сезонные дивизионы | да — сезонный MMR с нуля → Bronze…Master (пороги в аудите §7) |
| Награды за дивизионы | ешки + сезонная лимитка + сезонный титул по финальному дивизиону |

Двойной учёт: **lifetime MMR** (архив, для истории) + **season MMR** (сбрасывается,
драйвит дивизионы и награды).

## 6. Ежедневная мотивация (новые системы)

| Система | Механика | Сток/Mint | Приоритет |
|---|---|---|---|
| 🎁 Daily reward | заход раз в день → награда по дню серии (1..7 цикл), ~10→30 | контролируемый mint | 🔴 |
| 📋 Weekly missions | 3–5 заданий/неделя (открой N кейсов, ферми X раз, выиграй дуэль) → ешки + сезонные очки | mint + sink-драйвер | 🔴 |
| 🗓 Сезонные задания | длинные цели на сезон (открой 100 кейсов, дойди до Gold) → лимитка/титул | разово | 🔴 |
| 🔗 Цепочки активности | стрик-бонусы фермы + daily (пропуск = сброс) | множитель, не новый mint | 🔴 |

Калибровка: суммарный daily+weekly mint ≤ ~60/день, чтобы потолок честной игры
оставался ~145/день (§2). Missions в первую очередь гонят игрока в **стоки**
(кейсы), а не печатают ешки.

## 10. Списки на реализацию

### Миграции (Alembic)
1. `season` (id, name, started_at, ends_at, is_active).
2. `season_mmr` — сезонный MMR (поле в `users` или журнал `season_mmr_entries`).
3. `season_titles` (user_id, season_id, code, awarded_at).
4. `achievements`: добавить `season_id` + прогресс-таблицу при необходимости.
5. `login_streak` (user_id, last_login_date, current_streak, best_streak).
6. `daily_rewards` / `weekly_missions` / `season_quests` (определения + прогресс).
7. `season_archive_*` — архивные снапшоты (balance/mmr/leaderboards/inventory).
8. Сид сезонного кейса + сезонных лимиток.

### Изменения БД (данные, вайп)
- Снапшот → архив, TRUNCATE игровых журналов, обнуление полей `users`,
  сброс счётчиков `gift_catalog` (см. §1).

### Изменения бота (код)
- `app/settings/mmr.py` — анти-дуэль-фарм, пороги дивизионов, season MMR.
- `app/settings/balance.py` — лимиты передачи ешек, (опц.) события фермы.
- `app/settings/achievements.py` — новые категории + season_id.
- `app/features/farm/*` — анти-мультиаккаунт порог, связка со стриком.
- `app/features/gifts/*` — дневной лимит передач ешек.
- `app/features/duel/*` — лимит MMR-за-участие, разные оппоненты.
- Новые модули: `daily`, `missions`, `season` (старт/финал, награды, дивизионы).
- Админка: страница `/admin/season` (старт/стоп/награды) + контроль вайпа.

### Пошаговый запуск
1. Объявить дату вайпа, заморозка необратимых наград.
2. Снапшот в `season_archive_*`.
3. Применить миграции (season, дивизионы, login_streak, daily/missions, лимиты).
4. Включить антиабуз (§8).
5. Засидить сезонный кейс/лимитки/витрину, daily/weekly/quests.
6. Включить новые ачивки + season_id.
7. Выполнить вайп (§1).
8. Старт `season` (Сезон 1, 56 дней), объявить дивизионы/награды.
9. Мониторинг 72ч: `/admin/economy` (инфляция, перелив), `/admin/cases` (RTP).
10. Финал (56 дней): заморозить рейтинги, выдать сезонные награды, архив, Сезон 2.

## Главный принцип устойчивости

«Второй вайп через 2 месяца не нужен», если выполнены **3 условия**:
1. Антиабуз §8 закрыт (мультиаккаунт, перелив ешек/MMR, дуэль-фарм).
2. Mint (ферма+daily+weekly) откалиброван под потолок ~145/день, sink (кейсы/
   магазин) сильнее у активных тратящих.
3. Сезонный сброс снимает накопленный дисбаланс раз в 56 дней штатно — это
   плановый механизм, а не аварийный вайп.
