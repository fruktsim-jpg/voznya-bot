# Release 2.1 — последствия аpgrade aiogram 3.28 + недостающая web-часть

Узкая ревизия по двум темам из фидбэка. Не аудит всего проекта.

---

## Часть A. aiogram 3.13 → 3.28: что меняется

### Важная находка (блокер пересборки)
`requirements.txt` пинил `aiogram==3.13.1`, а `Dockerfile` ставит зависимости
из него (`pip install -r requirements.txt`). Значит **следующая пересборка
образа молча откатила бы сервер обратно на 3.13.1** и авто-выдача снова бы
отвалилась (`send_gift` → `send_gift_unsupported`). Исправлено: пин поднят до
`aiogram>=3.28,<3.29`. Перед деплоем сверь точную версию на сервере
(`pip show aiogram`) и при желании запинь её жёстко.

### Ответы на вопросы

**1. Работает ли теперь авто-выдача через send_gift без pending/manual?**
Технически — да, метод `bot.send_gift` в 3.28 есть, и адаптер
`app/services/telegram_gifts.py` его вызовет. НО фактически авто-выдача
включается только при выполнении ВСЕХ условий:
- `GIFTS_DELIVERY_ENABLED=true` (сейчас `false` — см. вопрос 6);
- у позиции каталога задан `telegram_gift_id` (иначе `no_telegram_gift_id`);
- на балансе бота хватает Stars (`getMyStarBalance >= star_cost`).
Если что-то из этого не так — доставка корректно остаётся `pending`, ешки не
теряются. То есть код 3.28-ready, но «само» не включится без флага + Stars +
gift_id.

**2. Нужно ли держать логику «доставка отключена / выдача вручную»?**
Да, оставить. Это не костыль 3.13, а штатные предохранители:
- ручная выдача (`/gifts_done`) нужна для **Premium** (его `sendGift` не
  выдаёт — см. вопрос 4) и как fallback, когда у бота кончились Stars;
- флаг `GIFTS_DELIVERY_ENABLED` — рубильник безопасности, чтобы случайно не
  начать жечь реальные Stars.
Удалять «manual/pending» нельзя — иначе Premium и сбойные выдачи зависнут.

**3. Менять ли цикл кейс → pending → ручная выдача на кейс → авто-выдача?**
Частично. Цикл «кейс → инвентарь/pending → решение игрока» остаётся базовым
(он нужен для P1/P5 — продать/оставить). Авто-выдача — это про то, что
происходит ПОСЛЕ «Оставить»/после покупки: обычный Telegram Gift при
`enabled=true` уедет автоматически (`deliver_gift` уже вызывается в боте сразу
после покупки). Для кейсовых призов авто-deliver сейчас НЕ вызывается в момент
открытия (по дизайну — игрок сначала выбирает «оставить/продать»). После
выбора «Оставить» можно добавить попытку `deliver_gift` — это маленькое
улучшение, не ломающее модель. Premium всё равно идёт через pending.

**4. Premium — ручной или автоматизированный?**
Остаётся ручным. `sendGift` отправляет только обычные Telegram Gifts из
`getAvailableGifts`. Telegram Premium как подписку Bot API через `sendGift`
не выдаёт (это отдельный продукт/подарок подписки `giftPremiumSubscription`,
который в aiogram 3.28 если и есть, то отдельным методом и со своими
ограничениями). Безопасный дефолт: Premium → `pending` → админ выдаёт вручную
через `/gifts_done`. Авто-путь для Premium — отдельная задача, не часть 2.1.

**5. Код telegram_gifts.py, ставший временным костылём эпохи 3.13.**
Проверил весь модуль. Кандидаты на чистку ПОСЛЕ подтверждения 3.28 на проде:
- `send_gift`: ветка `send = getattr(bot, "send_gift", None); if send is None:
  return ... "send_gift_unsupported"` (строки ~135-139) — защита от отсутствия
  метода в 3.13. В 3.28 метод есть всегда; можно звать `bot.send_gift(...)`
  напрямую. Оставить `getattr` безопасно, но это уже легаси.
- Аналогичный `getattr(bot, "get_my_star_balance", None)` и
  `getattr(bot, "get_available_gifts", None)` (строки 58-59, 78-79) — те же
  защиты «метод может отсутствовать». В 3.28 не нужны, но безвредны.
- Текст в `app/features/gifts/handlers.py`: `DELIVERY_REASONS[
  "send_gift_unsupported"]` («нужен aiogram 3.14+») — устаревшая подсказка,
  после апгрейда эта причина практически недостижима.
Рекомендация: НЕ удалять разом. `getattr`-проверки — дешёвая защита; убрать их
стоит только когда версия жёстко запинена и подтверждена на сервере. Это
вопрос косметики, не функциональности.

**6. Включён ли сейчас GIFTS_DELIVERY_ENABLED / реальная авто-доставка?**
НЕТ. В `app/config.py` дефолт `gifts_delivery_enabled = False`, в
`.env.example` — `GIFTS_DELIVERY_ENABLED=false`. Реального `.env` в репозитории
нет (значение задаётся на сервере). **Пока флаг не выставлен в `true` на
сервере — мы по-прежнему работаем в ручном режиме**, несмотря на апгрейд
aiogram. Чтобы включить авто-выдачу: на сервере `GIFTS_DELIVERY_ENABLED=true`,
пополнить Stars бота, проставить `telegram_gift_id` позициям каталога
(`/gifts_setid`). Проверить готовность: `/gifts_available` покажет баланс Stars
и доступные gift_id.

### Итог по A
Один обязательный фикс уже сделан (пин aiogram). Авто-выдача обычных Gifts
готова, но выключена флагом. Premium и ручной fallback остаются нужны.

## Часть B. Недостающая web-часть Release 2.1 (v0-voznya)

### Ключевой факт об архитектуре (исправляю прошлую заметку)
Сайт **не ходит** во внутренний API бота. `lib/cases-open.ts` — точный TS-порт
`open_case`, работающий НАПРЯМУЮ с общей Postgres в одной транзакции
(`withTransaction`), потому что сайт на Vercel, а бот на отдельном VPS, и
дотянуться до `internal_api.py` сайт не может. БД общая.

**Вывод:** web-продажу и web-вывод нужно делать так же — TS-логикой против БД
(новые `lib/gifts-sell.ts` / `lib/inventory-actions.ts` + route-хендлеры),
**а не** через эндпоинты в `app/web/internal_api.py`. Бот остаётся местом
ручной выдачи pending (Premium, fallback).

### Что есть сейчас на сайте
- `/cases` — открытие кейсов (`POST /api/cases/open` → `openCase`), уже
  возвращает `deliveryKey`, `starCost`, `rewardKind` для tg_gift.
- `/gifts` — read-only витрина коллекции (`getShowcaseGifts`). Покупка — только
  в боте; продажи/вывода нет.
- `/api/me/summary` — баланс/ранг для шапки (read-only).
- Полноценного раздела «Инвентарь» на сайте НЕТ (только админский
  `/api/admin/inventory`).

### Что нужно дозакрыть для Release 2.1 на сайте

**P1 (web) — экран выбора после открытия кейса.**
В UI результата открытия (bottom-sheet) для `rewardKind==='tg_gift'` показать
стоимость и кнопки «Оставить / Продать за N / Вывести». `deliveryKey` уже
приходит из `/api/cases/open`. Нужен только фронт + вызовы новых ручек ниже.

**P2 — раздел Инвентарь (`/inventory`).**
Нет вообще. Нужно:
- `lib/inventory-list.ts` (server-only): читает стековый `inventory` +
  `inventory_items`, и pending-`gift_transactions` игрока (его невыданные
  Gifts/Premium как «предметы во владении»).
- `app/inventory/page.tsx`: карточки предметов (визуалы уже есть в
  `lib/inventory.ts` — rarity/type), Premium отдельным блоком (статус «не
  активирован», действия «Активировать / Продать / Подарить» — активация может
  пока вести к ручному процессу).
- Источник предметов = кейсы + покупки магазина + будущее (как в ТЗ).

**P3 (web) — Магазин вместо Подарков.**
Переименовать раздел `/gifts` → `/shop` (или оставить путь, сменить заголовок и
навигацию «Подарки → Магазин»), добавить покупку за ешки прямо на сайте
(сейчас покупка только в боте). Покупка должна вести в инвентарь (pending), а
не в моментальную доставку.

**P4 — эндпоинты продажи/вывода (СЕЙЧАС ОТСУТСТВУЮТ на сайте).**
Добавить по образцу `/api/cases/open`:
- `POST /api/inventory/sell` — тело `{ deliveryKey }` (или inventory id), порт
  логики `sell_gift`: под `FOR UPDATE` строки `gift_transactions`, начислить
  70% (`ITEM_SELL_RATE`), перевести в `cancelled`, освободить резерв для
  покупок. Вернуть новый баланс.
- `POST /api/inventory/withdraw` — пометить предмет к выдаче: для pending-Gift
  это просто «оставить как есть» (он уже pending → попадёт в `/gifts_pending`
  бота); для предмета магазина — создать pending `gift_transaction`. Бот выдаёт.
- (опц.) `POST /api/shop/buy` — порт `buy_gift`, если покупку выносим на сайт.
Все — server-only, против общей БД, владелец из `session.uid` (не из тела).

**P6 — мгновенное обновление баланса.**
Сейчас баланс в шапке — `/api/me/summary`, тянется при загрузке. Нужно после
открытия кейса / продажи / покупки делать invalidate/refetch этого запроса
(React Query `queryClient.invalidateQueries(['me-summary'])` или ручной
refetch). Достаточно cache-invalidate, websocket не требуется для текущего
масштаба (как и сказано в ТЗ).

### Целевой цикл на сайте
```
Кейс (/api/cases/open) → предмет в Инвентаре (pending gift / stack item)
   → игрок в /inventory жмёт Продать (/api/inventory/sell)
        → +ешки в transactions, доставка cancelled
        → invalidate me-summary → баланс в шапке обновился
   → либо Вывести (/api/inventory/withdraw)
        → pending для бота → ручная/авто выдача
Магазин (/api/shop/buy) → тот же Инвентарь → те же действия
```

### Приоритет реализации web
1. P4 ручки `sell`/`withdraw` (бэкенд-логика, переиспользуется везде).
2. P2 `/inventory` (точка владения — без неё цикл не замкнут).
3. P1 (web) кнопки в результате открытия кейса.
4. P6 invalidate баланса (дешёво, цепляется к каждому действию).
5. P3 переименование + web-покупка (можно последним — покупка пока в боте).

> Замечание: вся серверная экономика (формулы 70%, компенсация возврата,
> атомарность) уже реализована и покрыта тестами в боте; web-часть — это
> TS-порт тех же правил против общей БД + UI. Дублирование логики неизбежно
> из-за split Vercel/VPS, поэтому новые TS-ручки обязаны 1:1 повторять
> `sell_gift`/`_sell_value`/`ITEM_SELL_RATE`.
