## 2. Аудит `inventory_items.ref_value`

- Колонка добавлена миграцией `0019_item_ref_value` (BIGINT NULL, CHECK ≥ 0).
- **Сейчас NULL у всех предметов** — оценочная стоимость ещё не проставлена.
- Влияние: полный EV кейсов считается только по валютным наградам; предметные
  награды без `ref_value` помечаются «?» на странице аналитики кейсов.
- Текущий единственный живой кейс — `case_vagabond` (сид 0017) — **дропает только
  ешки**, предметов в нём нет. Значит его EV уже полный и точный без `ref_value`.
  `ref_value` понадобится, когда в кейсы добавят косметику/предметы.

### Рекомендация по значениям `ref_value`
- Привязывать к той же сетке: стоимость предмета в ешках ≈ его «справедливая»
  ценность. Для статусной косметики — по редкости (common→legendary).
- Это **справочное** поле для аналитики/EV, НЕ цена покупки и НЕ трогает баланс.
- Проставлять при заведении предмета в каталог (сейчас каталог косметики пуст —
  проставлять будет нечего, пока не появятся предметы).

## 3. Первый набор данных для тестирования экономики

Принцип: **данные, не схема.** Реализовано сид-миграцией `0020_seed_gift_catalog`
(идемпотентно, по `code`), чтобы набор был воспроизводим на любой среде и
откатывался. Никаких новых таблиц.

Набор по сетке ECONOMY_V2 §3 (1★≈10 ешек, наценка +20–50%):

| code | name | star_cost | price_eshki | наценка | stock | тир |
|---|---|---|---|---|---|---|
| `gift_heart` | Сердечко | 15 | 200 | +33% | 50 | дешёвый |
| `gift_bear` | Мишка | 15 | 200 | +33% | 50 | дешёвый |
| `gift_rose` | Роза | 25 | 320 | +28% | 30 | средний |
| `gift_rocket` | Ракета | 50 | 700 | +40% | 15 | дорогой |
| `gift_diamond` | Бриллиант | 100 | 1450 | +45% | 5 | премиум |

Почему такие числа:
- При ~45 ешек/день дешёвый Gift (200) = **~4–5 дней** грайнда — здоровый коридор
  «3–10 дней» из §3 V2.
- Наценка растёт с тиром (+33%…+45%) — дорогие Gifts дают больше стока на единицу.
- `stock` ограничивает расход Stars при всплеске (потолок: 50·15 + 50·15 + 30·25
  + 15·50 + 5·100 = **3 800 Stars** суммарного пула, если раскупят всё).
- `telegram_gift_id` оставлен NULL (авто-отправка — поток следующего этапа).
- `is_active=true`, `sort_order` по возрастанию цены.

Это даёт Economic Control Center и будущему магазину **реальный ассортимент** для
проверки: витрина, себестоимость, P&L-болванка (нули продаж до запуска покупки).

> Реальные продажи/выручка появятся только после реализации потока покупки
> (L3 из `ECONOMY_LOGGING_AUDIT.md`) — это следующий этап (Gifts-магазин).
